# Nero 虚拟智能体 (Virtual Agent) 需求规格说明书

## 1. 项目概述

- **项目名称：** Nero (虚拟智能体)
- **核心定位：** 一个拥有动态情绪反馈、自主任务执行能力（Agentic Workflow）且高度可扩展的 Python 化虚拟伙伴。
- **设计哲学：** “有温度的实干家”。既能提供情绪价值，又能进入沙箱执行复杂的系统任务。

------

## 2. 核心功能需求

### 2.1 情绪感知与动态人设 (Emotional Persona)

Nero 不再是静态的对话机器人，其性格会随互动而演变。

- **情绪状态机：**
  - **维度：** 采用 Valence（正负情绪）和 Arousal（激动程度）双轴模型。
  - **更新机制：** 实时分析过去 5-10 轮对话的情感极性，计算累积加权值。
  - **表现联动：** 情绪分值动态注入 System Prompt，改变 Nero 的回复语气（如：傲娇、专业、温顺、抵触）。
- **短/长期记忆：**
  - **短期：** 维护基于 Window 的对话上下文。
  - **长期：** 自动提取用户偏好、重要事实（如用户正在开发的项目、习惯的代码风格），存入向量数据库。

### 2.2 自主 Agent 执行能力 (Autonomous Execution)

参考 **Claude Code** 与 **OpenHands** 的逻辑，实现“感知-决策-行动-观测”的闭环。

- **任务编排：** 面对复杂指令（如“优化这个 Python 脚本的并发处理”），Agent 需先生成分步计划（Plan）。
- **工具箱 (Tools)：**
  - **FileSystem：** 读、写、检索本地文件（支持基于 `diff` 的增量修改）。
  - **Terminal：** 在隔离环境中执行 Shell 命令、运行测试。
  - **MCP 兼容：** 支持 **Model Context Protocol**，实现插件式能力扩展。
- **自我修正 (Self-Correction)：** 若命令执行报错，Agent 需自动分析 stderr 并尝试修复方案，直至任务成功或触发人工干预。

## 3. 技术架构方案 (Python Stack)

### 3.1 逻辑架构图

Code snippet

```
graph TD
    User((用户)) --> API[FastAPI Gateway]
    API --> Controller[Agent Controller - LangGraph]
    Controller --> Sentiment[情绪分析中间件]
    Controller --> Memory[记忆管理: Redis/VectorDB]
    Controller --> LLM[LLM Orchestrator: GPT-4o/DeepSeek]
    LLM --> Tools[工具执行器]
    Tools --> Sandbox[Docker/Conda Sandbox]
    Sandbox --> Filesystem[(文件系统)]
```

### 3.2 关键技术选型

- **核心框架：** `LangGraph` (用于构建有环状态机，实现自主迭代逻辑)。
- **数据模型：** `Pydantic AI` (利用类型安全特性，严格约束工具调用和模型输出)。
- **情绪分析：** `TextBlob` 或调用轻量级模型（如 `Qwen-1.5B`）进行实时情感打分。
- **异步通讯：** `FastAPI` + `WebSockets` (确保文字流式输出与虚拟形象动作指令的低延迟)。
- **安全隔离：** `Docker SDK for Python` (将所有的代码执行限定在容器内)。

## 4. 详细模块设计

### 4.1 情绪引擎 (The "Heart")

> **技术实现逻辑：**
>
> 1. 每轮对话结束后，后端异步计算 `sentiment_score`。
> 2. `sentiment_score` 影响 `Nero_State`（数值范围 -1 到 1）。
> 3. 下一次对话时，System Prompt 会动态追加：`"Nero当前心情：[愉快]。请以轻快、乐于助人的语气回答，并适当使用颜文字。"`

### 4.2 执行引擎 (The "Muscle")

> **参考 Claude Code 的操作规范：**
>
> - **安全性：** 禁止执行 `rm -rf /` 等高危命令，建立指令黑名单。
> - **反思环：** Agent 每次执行工具后，必须回答“我看到了什么”和“下一步我该做什么”。
> - **上下文压缩：** 面对长代码或长日志，使用 `Map-Reduce` 策略提取关键报错信息。

------

## 5. 扩展性规划 (Future Extensions)

- **多模态增强：** 接入 `Whisper` (语音输入) 与 `Edge-TTS` (语音输出)，并根据情绪标签触发不同的声音音色。
- **IDE 集成：** 作为 VS Code 插件运行，直接通过 LSP（语言服务器协议）获取代码上下文。
- **视觉实体：** 集成 `Live2D` 或 `Unity` WebGL 渲染，后端通过 WebSocket 推送表情索引（如 `{"action": "nod", "expression": "happy"}`）。

------

## 6. 非功能性需求

- **性能：** Agent 思考过程应透明化，实时向前端推送“思考日志”。
- **隐私：** 长期记忆支持本地化部署（使用 `ChromaDB` 或 `SQLite`）。
- **鲁棒性：** 即使 LLM API 调用失败，情绪状态机也应能保持稳定，不丢失当前上下文。
