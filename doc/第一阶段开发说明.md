# 第一阶段开发说明（问答 · 短期记忆 · 心情）

本文档在《需求文档》《开发文档》基础上，将**阶段一**落地为与**当前仓库实现**一致的产品与技术规格，便于排期、实现与验收。

---

## 1. 阶段定位

### 1.1 本阶段目标

| 维度 | 目标 |
|------|------|
| **问答** | 用户通过 Web 与角色实时对话；回复**流式**输出。 |
| **简单记忆** | 单次会话内维护**滑动窗口**对话上下文；可选按 `session_id` 文件持久化（见 `SESSION_STORAGE_MODEL`），**不做**向量检索与长期 RAG（归入第三阶段）。 |
| **心情** | 每轮由**评判 LLM** 结合 `character_traits.json` 与对话输出 `valence` 等，映射为 **心情指数 0～100**；经 **mood_injection.txt** 注入主模型；前端展示 **心情指数** 与可选 **label**。**不使用**跨轮 EMA 情绪累积。 |

### 1.2 明确不在本阶段的内容

- LangGraph ReAct、文件/终端工具、Docker 沙箱（**阶段二**）。
- 向量数据库、自动摘要写入长期记忆、复杂召回（**阶段三**）。
- Live2D、TTS/ASR（**阶段四**）。

### 1.3 与上游文档的对应关系

- **需求文档 2.1**：单轴心情 + 评判 LLM + 主模型演绎；短期记忆已实现，长期记忆未实现。
- **开发文档 §1**：FastAPI + WebSocket、流式、`system_base` + `mood_injection` + 历史；**无** γ 衰减与三档 Emotion Mapper。

## 2. 总体架构

### 2.1 逻辑结构

```text
浏览器（聊天 UI + 心情指数）
        │  WebSocket（JSON：session / emotion / token / done / error）
        ▼
FastAPI
  ├── 连接与会话（session_id）
  ├── 短期记忆：窗口（+ 可选文件持久化）
  ├── 心情：评判 LLM → valence/confidence/label → mood_pct（0～100）
  └── 主 LLM：System Base + Emotional Context + History → 流式生成
```

### 2.2 技术栈（阶段一最小集）

| 层级 | 选型 | 说明 |
|------|------|------|
| API | FastAPI | `/`、`/ws`、`/health`。 |
| 实时通道 | WebSocket | 同连接推送 session、emotion、token、done、error。 |
| 主对话 | OpenAI 兼容流式 API | `LLMClient.stream_chat`。 |
| 心情评判 | 同端点非流式 `complete_chat` | 输出严格 JSON，失败则 `mood_pct=50`。 |
| 轨迹（可选） | 本地 JSONL | `MOOD_TRACE_DIR`。 |

## 3. 功能规格

### 3.1 问答（实时对话 + 流式）

**行为**

- 用户发送 `type: chat` 与 `content`；服务端先 **评判心情**（若开启），再写入本轮用户消息至记忆，组装 messages，**流式**返回助手内容。

**Prompt 三层（强制）**

1. **System Base**：`prompts/system_base.txt`。
2. **Emotional Context**：`build_mood_prompt_injection(mood_pct, label)`，模板来自 `mood_injection.txt`。
3. **Chat History**：窗口内多轮消息（含本轮用户句）。

**非功能**

- 评判超时：`MOOD_JUDGE_TIMEOUT_S`；主模型流式错误通过 `error` 帧返回并打日志。

---

### 3.2 简单记忆

- **滑动窗口**：`CHAT_WINDOW_SIZE`（按「轮」理解，实现为消息条数上限，见 `ShortTermMemory`）。
- **持久化**：`SESSION_STORAGE_MODEL=file` 时使用 `DATA_STORAGE_PATH` 下 JSON；否则仅进程内。

**不做**：向量检索、跨会话用户画像自动抽取。

---

### 3.3 心情（评判 → 指数 → 注入）

**状态**

- **无**全局 \(E_t\) 状态机；每轮独立计算 **mood_pct**。
- **映射**：`v_eff = valence × confidence`，`mood_pct = round(clamp((v_eff + 1) × 50, 0, 100))`（实现见 `judge.py`）。

**评判输入**

- `character_traits.json`（结构化 OCEAN 与行为提示，**手动维护**）。
- 最近 `MOOD_JUDGE_MAX_MESSAGES` 条历史 + 当前用户句。

**鲁棒性**

- 评判解析失败：`mood_pct=50`，`label` 标明失败；对话继续。
- `MOOD_USE_LLM_JUDGE=false`：指数恒 50，不注入评判概要。

**前端**

- WebSocket `emotion`：`mood_pct`、`session_id`、可选 `label`。

## 4. 接口与协议

### 4.1 WebSocket

**客户端 → 服务端**

| type | 字段 |
|------|------|
| `chat` | `content`（用户文本） |

**服务端 → 客户端**

| type | 说明 |
|------|------|
| `session` | `session_id` |
| `emotion` | `mood_pct`（0～100）、`session_id`、可选 `label` |
| `token` | `delta` 增量文本 |
| `done` | 本轮结束 |
| `error` | `message` 说明 |

### 4.2 REST

- `GET /health`：健康检查。

## 6. 配置项清单（第一阶段）

| 配置项 | 含义 | 默认 / 说明 |
|--------|------|-------------|
| `CHAT_WINDOW_SIZE` | 记忆窗口 | 15 |
| `MOOD_USE_LLM_JUDGE` | 是否启用评判 LLM | true |
| `MOOD_JUDGE_MAX_MESSAGES` | 参与评判的历史条数上限 | 12 |
| `MOOD_JUDGE_TIMEOUT_S` | 评判请求超时（秒） | 45 |
| `MOOD_TRACE_ENABLED` | 是否写 JSONL 轨迹 | true |
| `MOOD_TRACE_DIR` | 轨迹目录 | data/mood_trace |
| `LLM_*` | API 与模型名 | 见 `.env.example` |
| `DATA_STORAGE_PATH` | 会话文件等 | data |
| `SESSION_STORAGE_MODEL` | short / file | short |

**已移除（旧版）**：`EMOTTION_GAMMA`、三档 Emotion Mapper 与 EMA 公式。

## 7. 测试与验收标准

### 7.1 功能验收

- [ ] 浏览器建立 WebSocket，多轮对话，助手回复**流式**显示。
- [ ] 开启评判时，不同用户输入下 **mood_pct** 或 **label** 有合理变化；关闭评判时指数恒为 50。
- [ ] 评判失败时仍可完成对话，且前端/轨迹可见降级行为。
- [ ] `data/mood_trace/*.jsonl`（若开启）含 `mood_pct`、注入全文与对话内容。

### 7.2 非功能验收

- [ ] 多 `session_id` 记忆与连接互不串线。
- [ ] 错误通过 `error` 帧或日志可诊断，响应中不泄露密钥。

## 9. 风险与对策

| 风险 | 对策 |
|------|------|
| 评判与主聊两次调用延迟/费用 | 可配置超时；后续可考虑小模型评判。 |
| 人设与评判 JSON 不一致 | 改人设时同步改 `system_base.txt` 与 `character_traits.json`。 |
| 心情对台词影响弱 | 强化 `mood_injection.txt` 或调整评判提示；见产品侧讨论。 |
| WebSocket 断线 | 前端刷新重连，新 session。 |
