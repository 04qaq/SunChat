# Nero 第一阶段开发说明（问答 · 简单记忆 · 情绪）

本文档在《需求文档》《开发文档》基础上，将**阶段一**落地为可执行的产品与技术规格，便于排期、实现与验收。

---

## 1. 阶段定位

### 1.1 本阶段目标

| 维度 | 目标 |
|------|------|
| **问答** | 用户与 Nero 通过 Web 实时对话；回复以**流式**输出，降低主观延迟。 |
| **简单记忆** | 在单次会话内可靠维护**有限轮次**的对话上下文；可选地将会话持久化到本地（便于刷新后恢复），**不做**向量检索与长期 RAG（归入第三阶段）。 |
| **情绪** | 每轮根据用户输入更新**情绪数值**，通过 Prompt 影响语气；前端展示**心情标签**。 |

### 1.2 明确不在本阶段的内容（避免范围膨胀）

- LangGraph ReAct 循环、文件/终端工具、Docker 沙箱（**阶段二**）。
- 向量数据库、自动摘要写入长期记忆、复杂召回（**阶段三**）。
- Live2D、TTS/ASR（**阶段四**）。

### 1.3 与上游文档的对应关系

- **需求文档 2.1**：情绪状态机与「短/长期记忆」——本阶段实现**情绪连续更新 + 短期窗口记忆**；长期记忆仅预留接口或配置位，不实现 RAG。
- **开发文档 §1**：本阶段对齐 **FastAPI + WebSocket**、**流式输出**、**三层 Prompt**、**情绪衰减公式**、**Emotion Mapper**、**极简 Web 聊天窗 + 心情标签**。

## 2. 总体架构

### 2.1 逻辑结构

```text
浏览器 (聊天 UI + 心情展示)
        │  WebSocket (JSON 控制帧 + 文本/Token 流)
        ▼
FastAPI 应用
  ├── 连接与会话管理（session_id）
  ├── 记忆模块：对话窗口 + 可选持久化
  ├── 情绪中间件：用户句 → S_new → E_t → 标签/风格
  └── LLM 编排：组装 System / Emotional / History → 流式生成
```

### 2.2 推荐技术栈（与《开发文档》技术总览一致，阶段一最小集）

| 层级 | 建议选型 | 说明 |
|------|----------|------|
| API | FastAPI | 异步、WebSocket、OpenAPI 友好。 |
| 实时通道 | WebSocket | 一条连接承载：会话元数据、情绪更新、模型 token 流。 |
| 大模型调用 | OpenAI 兼容 API 或 DeepSeek 等 | 抽象为 `LLMClient.stream_chat(...)`，便于换端点。 |
| 情感极性 S_new | TextBlob（英文为主）或轻量中文情感 API / 小模型 | 若中英文混合，需统一约定一种实现并文档化；失败时**降级为 0**，不阻断对话。 |
| 短期记忆存储 | 进程内结构（开发）→ 可选 Redis（部署） | 第一阶段可用**内存 + 可选本地文件/SQLite** 满足「简单持久化」。 |

## 3. 功能规格

### 3.1 问答（实时对话 + 流式）

**行为**

- 用户发送一条用户消息；服务端将其写入**短期记忆**，更新**情绪**，再调用 LLM **流式**返回助手回复；流式片段通过 WebSocket 推送到前端拼接展示。
- 支持**中止生成**（可选 P1）：前端发 `cancel`，服务端关闭当前流。

**Prompt 三层结构（强制）**

1. **System Base**：Nero 的固定人设（性格、背景、回答习惯）。单独文件或配置维护，便于迭代。
2. **Emotional Context**：由当前情绪值与 Emotion Mapper 生成的**自然语言指令**（例如：语气轻快/冷淡、是否使用颜文字等）。
3. **Chat History**：按时间排序的多轮 `user` / `assistant` 消息列表，仅包含窗口内的内容（见 3.2）。

**非功能**

- 首 token 延迟尽量低；网络与模型错误需向前端返回**可区分**的错误码与简短说明。

---

### 3.2 简单记忆化

**本阶段「记忆」定义**：在模型上下文窗口内，提供**可控长度、可预测**的对话历史，避免无限膨胀；可选持久化同一会话。

**短期记忆（必做）**

- 使用**滑动窗口**：保留最近 **N** 轮完整对话（建议默认 **N = 10～20**，可配置）。
- 每条消息结构至少包含：`role`（`user` | `assistant`）、`content`、可选 `timestamp`。
- 超出窗口时：**丢弃最早**的轮次（FIFO），与《开发文档》中「上下文对话缓存」一致。

**持久化（可选，建议 P1 完成其一）**

- **方案 A**：按 `session_id` 将窗口序列化为 JSON 文件（适合本地开发与单机部署）。
- **方案 B**：SQLite 单表存消息行（`session_id`, `role`, `content`, `created_at`），启动时按窗口加载。

**明确不做**

- 不向 Prompt 注入「从向量库检索的背景信息」。
- 不做跨会话的自动用户画像抽取（可留钩子函数名，实现为空）。

---

### 3.3 情绪（数值连续 + 映射到语气 + UI 标签）

**与需求文档的关系**：需求提到 Valence / Arousal 双轴；**本阶段为降低复杂度，先实现单轴「情感极性」与《开发文档》中的衰减公式**，双轴可作为后续迭代项记录在「扩展」中。

**情绪状态**

- 用标量 **E_t ∈ [-1, 1]** 表示当前整体情绪倾向（偏负面～偏正面）。
- 初值建议 **E_0 = 0**（中性）。

**用户句情感极性 S_new**

- 对**当前用户输入**做一次情感分析，得到 **S_new ∈ [-1, 1]**。
- 实现须可替换：TextBlob、词典、或小模型；需记录**语言假设**（如主中文则选中文友好方案）。

**衰减与更新（与《开发文档》一致）**

\[
E_{t} = E_{t-1} \cdot \gamma + S_{new} \cdot (1 - \gamma)
\]

- **γ（gamma）**：衰减系数，建议默认 **0.7**（可配置）。
- 每轮用户发言后更新一次 **E_t**；助手回复不改变 **E_t**（避免重复计算），除非产品后续要求「对助手回复也做情感」——本阶段**不实现**。

**Emotion Mapper（必做）**

- 将 **E_t** 映射为：
  - **emotion_tag**：短字符串，供前端展示（如 `warm`、`cold`、`neutral`）。
  - **style_hint**：一两句注入 **Emotional Context** 的说明文字。

**建议默认分档（可调整）**

| 条件 | emotion_tag（示例） | 风格提示（示例方向） |
|------|---------------------|----------------------|
| E_t > 0.5 | `warm` / `enthusiastic` | 热情、多肯定、可适当活泼用语 |
| -0.5 ≤ E_t ≤ 0.5 | `neutral` | 平稳、专业 |
| E_t < -0.5 | `cold` / `sharp` | 简短、略带冷淡或「毒舌」分寸（需与人设一致） |

**鲁棒性（对齐需求文档 6 非功能）**

- 情感分析失败：**S_new = 0**，仍更新公式（主要保持 **E_{t-1}** 主导），并打日志。
- LLM 失败：**不重置** **E_t** 与短期记忆。

**前端**

- 在聊天区域旁或顶部展示当前 **emotion_tag**（及可选简短文案）；若 WebSocket 推送情绪更新，应在用户发送后、流式开始前即可刷新标签。

## 4. 接口与协议设计

### 4.1 WebSocket 建议消息类型

以下为逻辑协议，具体字段名可在实现时微调，但建议区分**控制消息**与**内容流**。

**客户端 → 服务端**

| type | 说明 |
|------|------|
| `chat` | 携带 `session_id`（若无则服务端新建）、`content`（用户文本）。 |
| `ping` | 保活（可选）。 |

**服务端 → 客户端**

| type | 说明 |
|------|------|
| `session` | 返回 `session_id`、服务端时间等。 |
| `emotion` | 推送 `E_t`、`emotion_tag`、`gamma`（可选）。 |
| `token` | 流式片段，`delta` 为增量文本。 |
| `done` | 本轮回复结束；可带 `usage` 元数据（可选）。 |
| `error` | 错误码与消息。 |

### 4.2 REST（可选）

- `GET /health`：健康检查。
- `POST /sessions`：若不用 WS 首包建会话，可用 REST 预先创建 `session_id`。

## 5. 目录与模块划分（建议）

```text
app/
  main.py              # FastAPI 实例、路由、WebSocket 挂载
  config.py            # 模型 API、gamma、窗口长度、路径
  memory/
    short_term.py      # 窗口与持久化适配器
  emotion/
    sentiment.py       # S_new 计算
    state.py             # E_t 更新与 Mapper
  llm/
    client.py          # 流式调用封装
  prompts/
    system_base.txt    # 或 .yaml
api/
  ws.py                # WebSocket 处理逻辑（可拆）
frontend/              # 极简静态页或 Vite+React，二选一
```

## 7. 测试与验收标准

### 7.1 功能验收

- [ ] 浏览器可建立 WebSocket，发送多轮消息，助手回复**流式**显示。
- [ ] 刷新页面后：若未做持久化，行为符合预期（会话丢失或明确提示）；若启用持久化，**同一 session_id** 可恢复历史。
- [ ] 连续发送偏正面/偏负面语句，**E_t** 与 **emotion_tag** 变化符合直觉；辱骂/赞美测试用例各不少于 2 条。
- [ ] 情感服务不可用时，对话仍可进行，情绪保持连续不出现崩溃。

### 7.2 非功能验收

- [ ] 并发多个 `session_id` 互不串记忆、不串情绪状态。
- [ ] 主要错误路径有日志，不在前端暴露密钥或完整堆栈。

## 9. 风险与对策

| 风险 | 对策 |
|------|------|
| 情感分析对中文不准 | 选用中文友好方案或混合策略；Mapper 分档放宽，避免标签剧烈抖动。 |
| 上下文过长导致费用上升 | 严格控制窗口轮次；System Base 保持精简。 |
| WebSocket 断线 | 前端重连 + 新 `session_id` 或恢复策略（与持久化方案一致）。 |
